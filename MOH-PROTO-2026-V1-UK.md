## MoH Proto (Message over HTTPS Protocol)
**RFC-стиль: мінімалістична специфікація протоколу — 2026 / Версія 1**

*Document ID:* `MOH-PROTO-2026-V1-UK`  
*Статус:* Draft  
*Транспорт:* HTTPS + Push-повідомлення

---

## 1. Анотація

MoH Proto (Message over HTTPS Protocol) визначає мінімалістичний стандартизований шар поверх HTTPS для двонапрямного обміну повідомленнями між Клієнтом і Сервером.

Протокол стандартизує:

- Кастомні HTTP-заголовки (`x-moh-*`)
- Процедуру хендшейку (HTTP `OPTIONS`)
- Інтерпретацію окремих HTTP статус-кодів
- Зворотні нотифікації Сервер → Клієнт через gateway URL та/або push-транспорт

---

## 2. Умовні позначення і терміни

Ключові слова **MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**, **MAY** інтерпретуються як у RFC 2119.

- **Клієнт**: ініціатор запиту MoH Proto (наприклад, мобільний застосунок або веб-застосунок).
- **Сервер**: відповідач MoH Proto.
- **Субʼєкт (`sub`)**: ідентичність сторони, представлена публічним ключем Ed25519 у нижньому регістрі hex (32 байти → 64 hex-символи).
- **Підпис (`sig`)**: підпис Ed25519 у нижньому регістрі hex (64 байти → 128 hex-символів).
- **Часова мітка (Timestamp)**: UTC Unix timestamp у мілісекундах, десятковим рядком.
- **Gateway**: наданий Клієнтом HTTPS endpoint для сервер→клієнт нотифікацій.
- **Root**: базовий URL, від якого резолвляться відносні посилання.

Якщо не вказано інакше, імена заголовків не чутливі до регістру (за правилами HTTP), але в прикладах використовується нижній регістр.

---

## 3. Транспорт

- **HTTPS** використовується для запитів Клієнт → Сервер та синхронних відповідей Сервер → Клієнт.
- **Push-повідомлення** використовуються для асинхронної доставки Сервер → Клієнт. MoH Proto стандартизує “конверт” нотифікації, але не стандартизує поля конкретного push-провайдера.

---

## 4. Реєстр заголовків (x-moh-*)

### 4.1 Спільні заголовки автентифікації

Клієнти і Сервери використовують такі заголовки для автентифікації повідомлень.

- **`x-moh-sub`**: публічний ключ субʼєкта, hex-encoded Ed25519 public key.
- **`x-moh-sig`**: підпис, hex-encoded Ed25519 signature.
- **`x-moh-timestamp`**: часова мітка (Unix ms) як рядок.

### 4.2 Заголовки хендшейку / можливостей

- **`x-moh-notify`** (Клієнт → Сервер, опціонально): HTTPS Gateway URL, куди Сервер SHOULD надсилати нотифікації для цього `sub`.
- **`x-moh-root`** (Сервер → Клієнт): базовий URL для резолву відносних `Location` та інших відносних посилань.
- **`x-moh-title`** (Сервер → Клієнт, опціонально): назва для відображення в UI Клієнта.
- **`x-moh-icon`** (Сервер → Клієнт, опціонально): URL іконки (абсолютний або відносний до `x-moh-root`).
- **`x-moh-color`** (Сервер → Клієнт, опціонально): колірна підказка UI (наприклад, `#RRGGBB`).
- **`x-moh-require`** (Сервер → Клієнт, опціонально): індикатор вимоги. Ця специфікація визначає значення `input`.

Клієнти, що віддають ці заголовки у прикладний код, SHOULD прибирати префікс `x-moh-` і повертати ключі в `camelCase` (наприклад, `x-moh-root` → `root`).

---

## 5. Криптографічний профіль

### 5.1 Алгоритми та кодування

- **Тип ключів**: пара Ed25519 public/private key.
- **Підпис**: Ed25519.
- **Кодування**: hex у нижньому регістрі, без префікса `0x`.

### 5.2 Канонічний рядок для підпису

Щоб прибрати неоднозначність між реалізаціями, підпис **MUST** обчислюватися над UTF-8 байтами наступного канонічного рядка:

```
MOH-PROTO/1
<timestamp>
<method>
<request-target>
<notify-url>
<body-sha256-hex>
```

Де:

- `<timestamp>` — точне значення `x-moh-timestamp`.
- `<method>` — HTTP-метод у верхньому регістрі (наприклад, `OPTIONS`, `GET`, `POST`).
- `<request-target>` — шлях + опціональний query як на дроті (наприклад, `/status?x=1`).
- `<notify-url>` — точне значення `x-moh-notify`, або порожній рядок якщо відсутній.
- `<body-sha256-hex>` — SHA-256 digest тіла HTTP-повідомлення у нижньому регістрі hex; для порожнього тіла — SHA-256 від порожнього масиву байтів.

### 5.3 Правила перевірки

- Отримувачі **MUST** перевіряти `x-moh-sig` відносно канонічного рядка та `x-moh-sub`.
- Отримувачі **MUST** відхиляти повідомлення з часовими мітками поза вікном, визначеним реалізацією (РЕКОМЕНДОВАНО: ±300 секунд), щоб зменшити replay.
- Реалізації **SHOULD** мати додатковий захист від replay (наприклад, облік nonce), якщо це можливо.

---

## 6. Хендшейк (Bootstrap авторизації)

### 6.1 Мета

Хендшейк дозволяє Серверу дізнатись (а) ідентичність Клієнта (`x-moh-sub`), (б) proof of possession (`x-moh-sig`), та опціонально (в) куди надсилати нотифікації (`x-moh-notify`). Сервер відповідає власною ідентичністю та UI-метаданими.

### 6.2 Запит

Клієнт **MUST** ініціювати хендшейк, відправивши HTTP `OPTIONS` на endpoint Сервера (РЕКОМЕНДОВАНО: той самий endpoint, що і для наступних запитів).

Клієнт **MUST** включити:

- `x-moh-sub`
- `x-moh-sig`
- `x-moh-timestamp`

Клієнт **MAY** включити:

- `x-moh-notify`

### 6.3 Відповідь

При успішному хендшейку Сервер **MUST** повернути:

- HTTP `200`
- `x-moh-sub` (ідентичність Сервера)
- `x-moh-sig`
- `x-moh-timestamp`
- `x-moh-root`

Сервер **MAY** повернути:

- `x-moh-title`
- `x-moh-icon`
- `x-moh-color`

Підпис Сервера **MUST** обчислюватися за тим самим канонічним рядком (Розділ 5.2) у контексті відповіді (метод `OPTIONS`, той самий request-target).

---

## 7. Семантика запит/відповідь

MoH Proto не примушує до одного HTTP-методу. Типово реалізації використовують `GET` для читання та `POST` для передачі `input`.

Значення `path` і корисного навантаження визначає прикладний рівень, але наведені нижче статус-коди стандартизовані.

---

## 8. Статус-коди

### 8.1 401 Unauthorized (Потрібна автентифікація)

Сервер повертає **401**, щоб показати, що Клієнт має надати:

- `x-moh-sub`
- `x-moh-sig`
- `x-moh-timestamp`

Клієнт, отримавши 401, **SHOULD** (пере)виконати хендшейк і потім повторити оригінальний запит, якщо це безпечно.

### 8.2 403 Forbidden з `x-moh-require: input` (Потрібні дані)

Сервер повертає:

- **403**
- `x-moh-require: input`

щоб показати, що Клієнт має передати додаткові дані `input` (визначені прикладним рівнем) і повторити запит.

Клієнт, отримавши таку відповідь, **SHOULD** показати UI для вводу та повторити запит із переданим `input`.

### 8.3 301 Moved Permanently з `Location` (Редірект)

Сервер повертає:

- **301**
- `Location: <uri>`

щоб попросити Клієнта зробити ще один запит. Якщо `Location` є відносним, Клієнт **MUST** резолвити його відносно `x-moh-root` (якщо доступний) або відносно URL оригінального запиту.

### 8.4 200 OK

**200** означає нормальну успішну відповідь.

---

## 9. Нотифікації (Сервер → Клієнт)

### 9.1 Gateway-нотифікації (HTTPS)

Якщо Клієнт надав `x-moh-notify` під час хендшейку, Сервер **SHOULD** використовувати його як Gateway endpoint для server→client нотифікацій.

Сервер відправляє HTTPS `POST` на Gateway з:

- `Content-Type: application/json`
- `x-moh-sub` (ідентичність Сервера)
- `x-moh-sig`
- `x-moh-timestamp`

Тіло (JSON):

```json
{
  "type": "moh.notify",
  "to": "<client-sub-hex>",
  "path": "/relative-or-absolute-path",
  "body": "optional string payload"
}
```

Підпис нотифікації **MUST** обчислюватися за канонічним рядком (Розділ 5.2), де:

- `<method>` = `POST`
- `<request-target>` = request-target запиту на Gateway
- `<notify-url>` = порожній рядок (gateway POST і є нотифікацією)
- `<body-sha256-hex>` = SHA-256 від JSON-байтів тіла

Клієнти **MUST** перевіряти підпис для gateway-нотифікацій.

### 9.2 Push-повідомлення

Для push-транспорту реалізації **MAY** доставляти:

- Повний підписаний JSON-конверт (Розділ 9.1), або
- Мінімальний push payload, що інструктує Клієнта підтягнути оновлення по HTTPS.

Ця специфікація стандартизує семантику підписаного “конверта”; поля конкретного push-провайдера поза межами документа.

## 10. Помилки

Нестандартизовані помилки визначаються прикладним рівнем. Сервери **SHOULD** повертати human-readable body за можливості.

---

## 11. Міркування безпеки

- **Replay-атаки**: валідуйте `timestamp` і обмежуйте вікно прийняття; розгляньте nonce-трекінг на `sub`.
- **Розсинхрон часу**: Клієнти SHOULD синхронізувати час пристрою; Сервери SHOULD дозволяти розумне вікно skew.
- **Компрометація ключа**: при компрометації приватного ключа зловмисник може імітувати Клієнта. Реалізації SHOULD підтримати ротацію ключів (поза межами V1).
- **Експозиція gateway**: публічний `x-moh-notify` збільшує attack surface. Gateway MUST верифікувати підписи та мати rate limit.
- **TLS**: весь HTTP трафік MUST бути по HTTPS з сучасною TLS-конфігурацією.

## 12. Додаток A: TypeScript-модель (Ненормативно)

Нижче — приклад можливого API клієнта/сервера.

```typescript
type MoHProtoRequest = {
  sub: string; // identity of the requester (decoded from hex)
  path: string;
  input?: string; // input data
};

type MoHProtoResponse = {
  sub: string; // identity of the responder (decoded from hex)
  status: "success" | "unauthorized" | "error" | "redirect" | "input";
  body?: string;
  location?: string;
  headers?: Readonly<Record<string, string>>; // x-moh-* headers without prefix, camelCase keys
  raw: Response;
};
```
